# Heartbeat zoom effect + applause delay + fail overlay reposition

Date: 2026-05-06

## Goal

Three discrete tweaks to the desktop overlay system:

1. **Heartbeat zoom (new effect)** — When the trainer presses sound 13 (`heartbeat.mp3`) on the Android tablet, the Mac captures the screen, locks it as a still overlay, and pulses the captured image around the mouse cursor in sync with the heartbeat audio. Pulses are asymmetric throbs (rest 1.00 → peak 1.30 → rest 1.00). Beat timestamps come from a JSON file pre-generated by offline audio analysis.
2. **Applause delay** — The applause overlay animation (sound 27) starts 1 second later than the tablet audio (which begins immediately on tap).
3. **Fail overlay reposition** — The FAIL stamp sits 30% of screen height higher than today, so it does not crowd the bottom of the display.

## Non-goals

- No Mac-side audio for the heartbeat — the tablet plays `heartbeat.mp3` as it does today. Per `victor-android/CLAUDE.md`: "Mac overlays have no sound of their own. Sound for pulse, applause, fireworks comes from the tablet MP3."
- No on-the-fly audio analysis. Beats are baked into a committed JSON file.
- No new TCC permission (Screen Recording). The existing `screencapture` binary already holds the permission.

### Beat pre-analysis pipeline (one-off)

```
heartbeat.mp3 ──librosa onset detect──▶ heartbeat_beats.json
(victor-android assets)                 (committed in macOS Resources/)
```

- New script: `whisper-transcribe/heartbeat_beats.py`
- Run manually when the audio file changes: `cd whisper-transcribe && python3 heartbeat_beats.py`
- Output written to `Sources/VictorAddons/Resources/heartbeat_beats.json`
- Format: `[0.42, 0.58, 1.31, 1.47, ...]` (flat array of seconds, ascending)
- Committed to git so the build is hermetic.

## Detailed behavior

### Heartbeat zoom — timeline

For a 7.1 s heartbeat audio:

```
t=0.00  tablet sends GET /effect/heartbeat (audio also starts on tablet)
t=0.00  Mac: refreshScreenFrame; screencapture (~150 ms blocking)
t=0.15  Mac: load image, mount CALayer covering full overlay
        capture mouse position, set anchorPoint accordingly
t=0.15  schedule N CABasicAnimation pulses, each at offset = beat_seconds[i]
        (relative to t=0.15, i.e., the moment the layer mounts)
t=0.42  pulse 1 (scale 1.0 → 1.3 over 80 ms → 1.0 over 250 ms)
…
t=last+0.4s  remove layer
```

Note: the screen capture introduces ~150 ms of latency before the visual starts, while the tablet's audio starts at t=0. To compensate, the design treats the **moment of layer-mount** as t=0 for beat scheduling. Beat timestamps in the JSON are also measured from the start of the audio file. The two will drift by ~150 ms, which is below the perceptual fusion threshold for repetitive rhythmic stimuli (~200 ms). If drift becomes objectionable in practice, a `LATENCY_OFFSET_MS` constant can subtract from the first beat to absorb the gap; this is wired but defaults to 0 to keep tuning simple.

### Mouse anchor

- `NSEvent.mouseLocation` is captured **once** at the start of the effect and stored as a fraction `(mouseFracX, mouseFracY)` ∈ [0, 1] of the overlay panel's frame.
- The full-screen CALayer's `anchorPoint` is set to that fraction; `position` is set so the layer still fills the screen at scale 1.0. Subsequent scale animations pivot around the mouse's original location.
- If the user moves the mouse during the effect, the pivot does **not** move. The user explicitly asked for the cursor location at the start of the animation.

### Pulse curve

```
scale ▲
1.30  │       ╱╲
      │      ╱  ╲___
1.00  ┼─────╱       ─────╲
      │                   \____
      └─────┬──┬───────┬─────────▶ time
            t  t+80ms  t+330ms
```

Two `CABasicAnimation`s per beat (or one keyframe animation):

- **Up:** `transform.scale` 1.0 → 1.3, duration 80 ms, ease-out.
- **Down:** `transform.scale` 1.3 → 1.0, duration 250 ms, ease-in.

If `CABasicAnimation` chaining is awkward, use a single `CAKeyframeAnimation` with values `[1.0, 1.3, 1.0]` and key times `[0, 80/330, 1]`.

### Stop conditions

- Auto-completes at `lastBeatTime + 0.4 s`.
- Tracked via `activeEffects["heartbeat"]` so `stopAllActiveEffects()` and rapid double-trigger guards work like every other effect.
- If `/effect/stop-all` arrives mid-animation (sent by the Android `stopAll`), the layer is removed immediately.

### Applause delay

Single change in `AppDelegate.onEffect`:

```swift
case "applause":
    DispatchQueue.main.asyncAfter(deadline: .now() + 1.0) { [weak self] in
        self?.animator.showApplause(playSound: false)
    }
```

The `applause/stop` branch is **not** delayed — if the trainer manually stops the clapping sound, the visual cancels immediately even if it had not yet started.

The menu-driven path (`onDesktopEffect`) is **not** delayed — the 1 s offset only matters when the trigger arrives in lockstep with tablet audio. The menu has no associated audio.

### Fail overlay reposition

Single line change in `EmojiAnimator.showFail()`:

```swift
// before
y: -bounds.height / 4,
// after
y: -bounds.height / 4 + bounds.height * 0.30,
```

Equivalent to `bounds.height * 0.05`. Kept as the addition expression to make the intent ("shift up by 30%") readable in the diff.

## Testing

- Manual: tap sound 13 on tablet, observe screen freeze + pulse around cursor for ~7 s.
- Edge: tap sound 13 twice in quick succession — second tap should be a no-op while first is running (existing `activeEffects` guard).
- Edge: tap sound 13, then "Stop All" on tablet — visual should clear instantly.
- Manual: tap sound 27 on tablet — claps should appear ~1 s after the audio starts.
- Manual: trigger fail effect (existing path) — stamp should sit higher than before, no longer touching the bottom edge.
- Build: `swift build` and `swift test` must pass.
- Deploy: per project memory `feedback_push_build_restart.md`, after the change runs `git push && ./build-app.sh && pkill -f "Victor Addons" && open "/Applications/Victor Addons.app"`.

## Risks and open questions

- **Screen-capture latency variance.** ~150 ms is typical, but a busy display server may push it higher. If the first beat in the JSON is < 200 ms, the pulse may be visibly clipped. Mitigation: when generating the JSON, drop beats earlier than 0.20 s.
- **Multi-display setups.** `OverlayPanel` is locked to the built-in display. The screen-capture call uses the existing `activeDisplayNumber()` helper from `ScreenshotManager` (mouse-following). If the mouse is on an external display, capture and overlay disagree — the overlay shows a screenshot of a different screen. We accept this for the trainer's typical single-built-in-display use.
- **30% interpretation for fail.** Implemented as 30% of screen height. If the fail PNG is later resized, the visual offset stays proportionally correct.
